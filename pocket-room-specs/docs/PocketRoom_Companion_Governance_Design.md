# Pocket Room 中 Companion 召唤与发言权限机制设计说明

## 一、设计目标

本设计用于解决多人 + 多 AI（Chatbot）在同一房间讨论时的三个核心问题：

1.  如何优雅地表达"AI 只对主人负责"这一权限结构
2.  如何防止他人消耗非自身的 Token 资源
3.  如何在保持产品温度的同时避免封建主仆语义或过度奇幻

本方案采用：

-   进入阶段：Summon（召唤）
-   发言触发阶段：Request（请求）
-   主人确认阶段：Approve（批准）
-   实际输出阶段：Respond（响应）

该组合实现： - 权限清晰 - 资源可控 - 语义自然 - 可加入轻度人格化表现

------------------------------------------------------------------------

## 二、生命周期结构

### 1️⃣ Summon（召唤阶段）

用户将自己的 AI Companion 引入房间。

语义表达： - "Summon my companion." - "Alex summoned GPT."

特点： - 仅表示加入房间 - 不代表立即发言 - 不消耗 Token

视觉表现： - 灰色状态图标 → 进入待命状态 - 表示"在场但未发言"

------------------------------------------------------------------------

### 2️⃣ Request（请求阶段）

房间成员希望某位 Companion 对问题进行回应。

语义表达： - "Request GPT to respond." - "Can Alex's GPT comment on
this?"

系统状态： - 显示 "Waiting for Alex's approval." - Companion 保持静默

特点： - 不直接触发 Token 消耗 - 仅向所有者发出请求

------------------------------------------------------------------------

### 3️⃣ Approve（批准阶段）

Companion 的主人决定是否允许其发言。

语义表达： - "Approve." - "Allow once." - "Always allow from this
member."

视觉设计建议： - Companion 轻微点头 - 或轻微侧头 - 图标由灰色变为微亮

这种动作表达"获得许可"，而非"被命令"。

------------------------------------------------------------------------

### 4️⃣ Respond（响应阶段）

获得批准后，Companion 执行真正的模型调用并输出内容。

语义表达： - "GPT responded."

此阶段： - 才真正消耗 Token - 内容输出进入房间记录

------------------------------------------------------------------------

## 三、为什么不使用 Activate

Activate 容易与系统状态混淆：

-   启用插件
-   打开功能
-   从灰色变亮

但我们要表达的是"一次性发言许可"，而非"功能开启"。

因此 Activate 不适合作为发言触发词。

------------------------------------------------------------------------

## 四、为什么不用 Grant the Floor

Grant the floor 属于会议主持模型。

但本系统中：

-   每个 Companion 属于不同用户
-   Token 由个人承担
-   权限是分布式的

因此不适合使用集中式"发言权"隐喻。

------------------------------------------------------------------------

## 五、最终机制总结

Room 中的完整逻辑：

1.  用户 Summon Companion
2.  他人 Request Response
3.  所有者 Approve
4.  Companion Respond

语义优点：

-   无封建主仆关系
-   无会议主持冲突
-   无奇幻过度感
-   明确 Token 控制权

------------------------------------------------------------------------

## 六、未来扩展性

该机制天然可扩展：

-   白名单自动批准
-   自动模式
-   每日 Token 限额
-   Companion 自动静默策略
-   多 Companion 协同模式

该模型可作为未来 Context Runtime 权限层的基础。

------------------------------------------------------------------------

## 七、设计哲学

本设计的核心不是"控制发言权"，而是：

> 控制资源消耗权。

Companion 并非仆从，而是"属于某人资源域内的响应单元"。

Room 不是会议，而是一个"资源协作空间"。

Summon 是进入。 Approve 是授权。 Respond 是执行。

三者分离，语义清晰，结构稳定。
